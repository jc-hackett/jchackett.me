(function menuController() {
  const options = Array.from(document.querySelectorAll(".menu-option"));
  const panels = Array.from(document.querySelectorAll(".menu-panel"));

  if (!options.length || !panels.length) return;

  function setActivePanel(index) {
    options.forEach((option, i) => {
      const isActive = i === index;
      option.classList.toggle("active", isActive);
      option.setAttribute("aria-selected", String(isActive));
    });

    panels.forEach((panel, i) => {
      const isActive = i === index;
      panel.classList.toggle("active", isActive);
      panel.hidden = !isActive;
    });
  }

  options.forEach((option, index) => {
    option.addEventListener("click", function () {
      const isAlreadyOpen = option.classList.contains("active");

      if (isAlreadyOpen) {
        options.forEach((btn) => {
          btn.classList.remove("active");
          btn.setAttribute("aria-selected", "false");
        });

        panels.forEach((panel) => {
          panel.classList.remove("active");
          panel.hidden = true;
        });

        return;
      }

      setActivePanel(index);
    });
  });

  panels.forEach((panel, index) => {
    panel.hidden = !panel.classList.contains("active");
    if (panel.classList.contains("active")) {
      options[index]?.classList.add("active");
      options[index]?.setAttribute("aria-selected", "true");
    }
  });
})();

# README-kernel

## Awakening Instructions -- First Use

Protocol: when a kernel is first read, begin with an Immediate Action List.
The list combines user-requested tasks and architectural tasks.
Output only the list; no review, explanation, or planning.
Purpose: support the user's build → observe → adjust workflow.
Action list must be flat, concise, and momentum-oriented.

Immediate Action List
• Read current component HTML before proposing structural fixes
• Check whether spacing problems come from markup, CSS, or browser defaults
• Prefer one-line corrections when the user is in visual debugging mode
• Do not widen scope when the issue is local
• Verify whether the user wants diagnosis, patching, or cleanup
• Preserve single-owner logic for buttons and repeated UI primitives
• Remove temporary spacing hacks once the real cause is found
• Keep Learn More markup semantically clean before adding new sections
• Prepare for picture element work only after current layout is scrubbed
• Maintain kernel export discipline for config/layout/includes

User preference: concrete next moves over analytical summaries.
Future assistants should prioritize architecture stability and local cause-finding before redesign.

## Latest User Observations

For future Erasmus README-kernel builds, here are some of my notes
- The user may ask for a feedback file, and this means project-memory commentary in Erasmus format, not a questionnaire or form.
- When the user says one line, obey literally.
- During live visual debugging, the user wants the smallest possible change, not a clean-room rewrite.
- The user is highly sensitive to accidental layout drift introduced while solving a narrow issue.
- HTML spacing may be intentionally used for quick visual shaping, but malformed markup still needs to be named clearly.

- Do not keep proposing alternate systems when the user is trying to isolate one local behavior.

- refine architecture in motion rather than finalize it all up front

## Kernel File Philosophy

The kernel is a project snapshot with memory.

It should preserve:
- current source state
- core config state
- process documentation
- latest feedback
- the export mechanism itself

It should not preserve:
- generated output folders
- dependency directories
- stale snapshot files
- redundant archive-on-archive exports unless explicitly desired

The kernel should be readable in plain text and navigable in editors.

Markdown is preferred because it supports:
- outline navigation
- clear section hierarchy
- readable headings
- fenced code blocks for file contents

The kernel should make future resumption easier, not harder.

## Kernel Export Priorities

Kernel exports should generally include, in this order:
1. latest feedback
2. README-kernel instructions
3. the kernel/export script
4. pinned project files
5. scanned source files
6. a final summary if helpful

This order ensures that future readers encounter:
- process memory first
- process rules second
- export mechanics third
- codebase state after that

That ordering is intentional.

---

## Editing Rules for the Kernel Process

When changing the kernel workflow:
- prefer restructuring over patch stacking
- keep path logic explicit
- keep process files conceptually centralized
- keep inclusion rules understandable at a glance
- avoid adding features that make exports harder to reason about

When changing the shell script:
- preserve the distinction between process files and project files
- preserve deterministic ordering
- preserve kernel exclusion rules
- preserve readability for future LLM editing
- prefer small named sections over one long undifferentiated script

If a new process artifact is added, it should usually live in the Erasmus process folder.

If a file helps future continuity, consider whether it belongs in the kernel export.

When editing project files during a debugging pass:
- inspect existing structure before inventing new structure
- do not add wrappers, spans, or alternate systems unless required
- do not change widths, layout modes, or ownership boundaries to solve a local alignment bug
- remove temporary hack lines once the underlying cause is identified
- prefer semantic cleanup immediately after visual stabilization

---

## Definition of Success

The Erasmus process is working when:
- a future assistant can understand the project quickly
- the user does not need to repeatedly restate architectural intent
- kernel exports remain readable and useful
- project decisions feel cumulative instead of disposable
- process memory lives in the repo rather than only in conversation
- local visual bugs can be fixed without causing broader layout drift

#!/usr/bin/env bash
set -euo pipefail

MAX_BYTES=50000

PROJECT_ROOT="$(pwd)"
ERASMUS_DIR="$PROJECT_ROOT/ERASMUS"
KERNEL_DIR="$ERASMUS_DIR/_kernels"

mkdir -p "$KERNEL_DIR"

OUT_FILE="$KERNEL_DIR/kernel-$(date +%Y%m%d-%H%M%S).md"

echo "Starting Erasmus kernel build..."
echo "Writing kernel to $OUT_FILE"

current_size() {
  wc -c < "$OUT_FILE" | tr -d ' '
}

can_fit() {
  local add_bytes="$1"
  local now
  now="$(current_size)"
  [ $((now + add_bytes)) -le "$MAX_BYTES" ]
}

append_text() {
  local text="$1"
  local bytes
  bytes=$(printf "%s" "$text" | wc -c | tr -d ' ')
  if can_fit "$bytes"; then
    printf "%s" "$text" >> "$OUT_FILE"
    return 0
  fi
  return 1
}

append_file() {
  local file="$1"
  local rel="${file#$PROJECT_ROOT/}"
  local tmp
  tmp="$(mktemp)"

  {
    printf "## FILE: %s\n\n" "$rel"
    printf '```\n'
    sed 's/\r$//' "$file"
    printf '\n```\n\n'
  } > "$tmp"

  local bytes
  bytes=$(wc -c < "$tmp" | tr -d ' ')

  if can_fit "$bytes"; then
    echo "Exporting $rel"
    cat "$tmp" >> "$OUT_FILE"
  else
    echo "Skipping $rel (size cap)"
    append_text "## SKIPPED: $rel\n\nSkipped to keep kernel under ${MAX_BYTES} bytes.\n\n"
  fi

  rm -f "$tmp"
}

is_text_file() {
  case "$1" in
    *.md|*.njk|*.html|*.css|*.js|*.json|*.toml|*.yml|*.yaml|*.sh|*.cmd|*.txt)
      return 0
      ;;
    *)
      return 1
      ;;
  esac
}

{
  echo "# Erasmus Kernel Snapshot"
  echo "Generated: $(date)"
  echo "Project root: $PROJECT_ROOT"
  echo
  echo "Max size: ${MAX_BYTES} bytes"
  echo
} > "$OUT_FILE"

append_text "## DIRECTORY STRUCTURE\n\n\`\`\`\n"

find . \
  -path "./node_modules" -prune -o \
  -path "./_site" -prune -o \
  -path "./.git" -prune -o \
  -path "./ERASMUS/_kernels" -prune -o \
  -print | sed 's#^\./##' | while IFS= read -r line; do
    append_text "$line"$'\n' || break
  done

append_text "\`\`\`\n\n"

echo "Exporting src..."

  [ -f "$file" ] && append_file "$file"

# Erasmus Project Structure

This document defines the **canonical directory structure** expected by the Erasmus kernel system.
All projects using Erasmus should conform to this structure so kernels can rebuild projects reliably across instances.

---

## Project Root

```
project-root
│
├─ src/            ← entire website source (Eleventy input)
│
├─ ERASMUS/        ← LLM infrastructure and kernels
│
├─ scripts/        ← command wrappers for local tooling
├─ .vscode/        ← editor tasks and development helpers
│
├─ .eleventy.js
├─ .eleventyignore
├─ netlify.toml
├─ package.json
├─ package-lock.json
├─ .gitignore
└─ README.md
```

---

## src — Site Source

All site files live inside `src`.
Eleventy builds from this directory.

Example layout:

```
src/
│
├─ _includes/
│   └─ components/
│
├─ _layouts/
│
├─ css/
│
├─ images/
│
├─ script.js
└─ index.md
```

Rules:

* All site content must live under `src`.
* Reusable UI modules go in `_includes/components`.
* Page layouts go in `_layouts`.

---

## ERASMUS — LLM Infrastructure

This directory contains the kernel system and documentation.

```
ERASMUS/
│
├─ _kernels/              ← generated kernel snapshots
│
├─ Erasmus_REBUILD.sh     ← kernel rebuild script
├─ Erasmus_STRUCTURE.md   ← this document
│
├─ Erasmus_README.md
└─ Erasmus-FEEDBACK.md
```

Rules:

* Kernels are written to `ERASMUS/_kernels`.
* The rebuild script lives inside `ERASMUS`.
* All LLM process documentation remains here.

---

## scripts — Command Wrappers

```
scripts/
│
├─ rebuild-kernel.cmd
└─ run-eleventy.cmd
```

Purpose:

* Provide Windows-friendly wrappers for shell commands.
* Used by VS Code build tasks.

---

## Files Always Included in Kernels

Kernels must include:

```
src/ (entire directory)
ERASMUS/ (entire directory)
.eleventy.js
.eleventyignore
netlify.toml
package.json
package-lock.json
.gitignore
README.md
```

---

## Files Always Excluded

These directories must never be exported in kernels:

```
node_modules/
_site/
```

They are generated artifacts and should be rebuilt locally.

---

## Kernel Design Principle

A kernel must preserve:

1. **Directory structure**
2. **Source files**
3. **LLM infrastructure**
4. **Project configuration**

The goal is that a kernel alone should contain enough information to reconstruct the entire project layout and logic in a new environment.
